A good webapp should **not simply digitise the ballot spreadsheet**. It should model the whole allocation lifecycle.

## Core process to build

### 1. Annual setup by staff

Staff create a new academic year and configure:

* UG / PG process calendars.
* Licence start dates.
* Ballot phases.
* Room stock.
* room eligibility: UG, PG, 4th-year-only, fresher-only, unavailable, refurbishment, temporary accommodation.
* rent bands and confirmed rents.
* household-size rules.
* special requirement categories.
* ballot priority rules.

This matches the timetable work currently done before Christmas: room stock planning, fair split between UG/PG, KX updates, rent band updates, special forms, notices and FAQs.

---

### 2. Student application phase

Students log in with Raven/SSO and complete the correct workflow:

#### Undergraduate

* confirm they need accommodation;
* form household;
* nominate representative;
* select household size;
* apply for triple set where relevant;
* declare if continuing as 4th year;
* special requirements handled separately.

UG process rules include 3rd years before 2nd years, a separate 4th-year ballot, group/double/single sequencing, and 30-minute household representative slots.

#### Postgraduate

* confirm accommodation need or opt out;
* form individual or group household;
* nominate representative;
* select household size;
* apply for furnished let / couples-family accommodation;
* apply to retain current room where eligible;
* special requirements handled separately.

PG has its own priority categories: third years, second years, internal first years, continuers, tenth termers, and post-tenth-termers.

---

### 3. Household formation

This should be an explicit system object, not just form text.

A **Household** would have:

* representative;
* members;
* programme type: UG / PG / mixed;
* academic year category;
* ballot household size;
* requested size;
* eligibility status;
* random ballot number;
* assigned time slot;
* chosen rooms;
* final allocation status.

The app should validate household rules automatically:

* mixed UG year groups use the mode; ties go to the lower-priority year group.
* PG mixed-year groups also use the mode, with even splits going to the lower year’s ballot.
* UG/PG mixed households require permission and are entered into PG ballot rules.
* PG small houses of up to 6 rooms can have retain-room rules; larger houses behave differently.

---

### 4. Special requirements / pre-allocation workflow

This should be a separate restricted-access workflow because it involves sensitive information.

Process:

1. student submits special requirement;
2. system records supporting evidence status, not necessarily the evidence itself unless policy allows;
3. College Nurse / authorised staff review;
4. staff approve/reject;
5. Accommodation allocates suitable room;
6. student is removed from main ballot.

Both UG and PG notices say students must re-declare each year, pre-allocation does not carry over, medical evidence may be required, late submissions are not considered, and successful applicants do not take part in the ballot.

This area needs strong permissions and audit logging.

---

### 5. Room list / availability engine

Instead of a live SharePoint spreadsheet, the app should have a live room inventory.

Room fields:

* room code/name;
* site/building/staircase;
* room type;
* rent;
* rent band;
* UG/PG eligibility;
* household grouping;
* ballot household size;
* overall household size;
* ensuite/private/shared facilities;
* availability status;
* refurbishment/noise notes;
* temporary accommodation flag;
* photos/floorplan link;
* current chosen occupant;
* allocation lock status.

The current process relies on view-only room lists updated live during the ballot. The notices explicitly say the Accommodation team updates the list with occupant names as rooms are chosen.

The app should replace that with transactional updates: once a room is selected, it is locked immediately.

---

### 6. Ballot generation

Staff should be able to generate ballot order after preferences close.

The system should:

* exclude pre-allocated students;
* validate all households;
* apply priority category;
* assign auditable random numbers;
* generate ordered ballot queues;
* generate 30-minute time slots;
* publish slots to students;
* email representatives.

The existing process uses random numbers and publishes slots before the ballot. Students are told their ranking is generated by an online random sequence process and that ballot slots are emailed/published.

Important: the app should store the random seed/input snapshot so the ballot can be audited later.

---

### 7. Live room selection

During a household’s slot:

* representative logs in;
* sees only eligible available rooms/households;
* can filter by rent, location, size, facilities;
* selects rooms;
* assigns named students to rooms;
* confirms;
* system locks allocation.

This replaces the current “email Accommodation within your 30-minute slot” process.

For missed slots:

* household remains eligible;
* can select later;
* reduced availability is visible;
* staff can override if needed.

Both UG and PG FAQs say missed slots do not remove the household, but the process continues and others may choose ahead of them.

---

### 8. Exception handling

The webapp needs explicit admin tools for awkward cases:

* no household of requested size remains;
* group splits into smaller households;
* group joins larger household;
* double oversubscription;
* triple set pre-allocation;
* 4th-year conditional continuation;
* PG post-10th-term handling;
* student intermits or does not return;
* room withdrawn after publication;
* student unhappy and joins waiting list.

UG rules around splitting groups and retaining household random numbers are detailed in the notices.

---

## Suggested system roles

* **Student**
* **Household representative**
* **Accommodation staff**
* **Student Services**
* **College Nurse / disability reviewer**
* **Senior Tutor / Director for Postgraduates approver**
* **Read-only JCR/SBR reviewer**
* **System administrator**

---

## MVP scope

For a first version, I would build:

1. Room inventory import from Excel.
2. UG/PG student import.
3. Student preference form.
4. Household creation and validation.
5. Special requirements tracking, but with limited sensitive detail.
6. Ballot generation.
7. Time-slot publication.
8. Live student room selection.
9. Staff override/admin panel.
10. Export back to KX / CSV.

Do **not** start with a fully automated allocation optimiser. The College process is rule-heavy, exception-heavy, and needs human discretion. The first win is replacing email + SharePoint + manual spreadsheet updates with a controlled workflow.

---

## Best architecture

Given your environment, I’d suggest:

* **Laravel backend**
* **SQL Server / Azure SQL database**
* **Microsoft Entra / Raven SSO if available**
* **role-based admin portal**
* **student-facing portal**
* **CSV/XLSX import/export**
* **audit log table for every allocation change**
* **queued email notifications**
* **Power Platform integration later, if needed**

The database model should be centred around:

* `academic_years`
* `students`
* `rooms`
* `room_groups`
* `applications`
* `households`
* `household_members`
* `special_requirement_requests`
* `pre_allocations`
* `ballot_phases`
* `ballot_entries`
* `ballot_slots`
* `allocations`
* `allocation_events`
* `waiting_list_entries`

The key design principle: **students should never be emailing choices into a shared inbox during a live ballot**. They should choose from live availability, and the system should make the resulting allocation atomic, auditable, and reversible only by staff.
